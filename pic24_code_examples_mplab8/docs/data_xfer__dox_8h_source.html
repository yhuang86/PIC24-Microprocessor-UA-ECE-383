29  * \page dataXfer The uC/PC data transfer protocol
30  * \section dataXferIntro The PC/uC data transfer protocol
31  *
32  * The uC data transfer protocol is a simple protocol to exchange data between
33  * a PC and a PIC over the UART. The example given below and implemented in
34  * this library applies to the PIC24/dsPIC33 series, though porting it to other
35  * microcontrollers is straightforward.
36  *
37  * \htmlonly
38  * <object width="640" height="505"><param name="movie" value="http://www.youtube.com/v/3B60xpMLoFg&hl=en_US&fs=1&"></param><param name="allowFullScreen" value="true"></param><param name="allowscriptaccess" value="always"></param><embed src="http://www.youtube.com/v/3B60xpMLoFg&hl=en_US&fs=1&" type="application/x-shockwave-flash" allowscriptaccess="always" allowfullscreen="true" width="640" height="505"></embed></object>
39  * \endhtmlonly
40  *
41  * \section usageSketch Usage sketch
42  * \code
43 #include "pic24_all.h"
44 #include "dataXfer.h"
45 
46 // Indexes of all the variables to be transferred.
47 enum { U16_NUMCHARS_NDX, C_NDX };
48 
49 // Number of characters to print on each line.
50 #define CHARS_PER_LINE 10
51 
52 int main(void) {
53  char c;
54  uint16_t u16_numChars = 0;
55 
56  // Initialize
57  configBasic(HELLO_MSG);
58  initDataXfer();
59 
60  // All variables received by the PIC must be specified.
61  // Params: Index Variable PC can change Format Description
62  SPECIFY_VAR(U16_NUMCHARS_NDX, u16_numChars, TRUE, "%hu", "Number of characters received");
63  SPECIFY_VAR(C_NDX, c, FALSE, "%c", "Character entered");
64 
65  while (1) {
66  // For debug support, send the index and char received.
67  sendVar(U16_NUMCHARS_NDX);
68  sendVar(C_NDX);
69 
70  // Echo a character
71  c = inCharXfer();
72  outChar(c);
73 
74  // Add a CR every CHARS_PER_LINE
75  if ((u16_numChars++ % CHARS_PER_LINE) == 0)
76  outChar('\n');
77  }
78 }
79  * \endcode
80  *
81  * \section dataXferExamples Examples
82  * The following programs use the data transfer protocol, providing working
83  * examples of its usage.
84  * - dataXfer_echo.c, which is given above.
85  * - dataXfer_demo.c, a more complete example.
86  * - ledpwm.c
87  *
88  * \section designGoals Design goals
89  * - Easy to use: no setup required to send variables; integrates with
90  * ISRs seamlessly by using getch/putch rather than direct assignment
91  * to UxRXREG/UxTXREG. Receive requires setup and returns a non-protocol
92  * character read or a variable read.
93  * - Excellent error reporting: PIC defines the protocol, pushing most
94  * errors onto the PC. Timeouts on the PIC and PC help spot errors. The
95  * PIC reports errors via text messages, which flow through to the PC.
96  * - Reasonably efficient (> 100 Hz). Two bytes of overhead per variable (start, type).
97  * - Minimal PIC memory requirements: one pointer, one uint8_t, one bit per variable.
98  * - Can send/receive data in any order (not a fixed sequence): sendVar
99  * in any order, receiveVar receives any var (or even a non-protocol
100  * character, to make interaction via a menu work)
101  *
102  * \section errorCases Error cases
103  * - Timeout during received of a variable. In this case, the variable
104  * is only partially assigned; the PIC is reset, while the PC reports an error.
105  * - Sending in an ISR, which could cause timing problems. Fixing this
106  * really needs a trace buffer implementation.
107  * - Index of variable not the same between PIC and PC. The PIC names
108  * variables; the PC likewise only allows access to variables of the same
109  * name. However, the PC code will enforce variable name to number mapping,
110  * making this fairly uncommon (if the same name occurs twice, there's a problem).
111  *
112  * \section protocol Protocol
113  * - \ref CMD_TOKEN varBits data
114  * - varBits = { v v v v v v l l} bitfield
115  * - v = variable number (0-\ref NUM_XFER_VARS, 63 = special)
116  * - l = length (1-4 bytes, encoded as 0-3)/special code. Special codes:
117  * - code = 0: escaped \ref CMD_TOKEN : this is the value \ref CMD_TOKEN,
118  * not the start of a command.
119  * - code = 1: long var
120  * - code = 2-3: Variable specification: data is size (1 byte),
121  * format string, name, description (all
122  * zero-terminated). Code 2: send only var (PC may not modify);
123  * code 3: send/receive var.
124  * - Only the PIC can send codes 2-3.
125  * - For all but code 0, data = varNum (1 byte), length (1 byte),
126  * data (of length bytes). varNum must be between 1 and
127  * \ref NUM_XFER_VARS.
128  * - For non-special, data is len bytes. For special: see above. All
129  * lengths are len + 1, e.g. len = 0 is 1 byte, etc.
130  * - Sending a \ref CMD_TOKEN, whether inside a protocol packet or as a
131  * character not in a packet, must always be escaped with a
132  * \ref CMD_TOKEN \ref ESCAPED_CMD.
133  */
